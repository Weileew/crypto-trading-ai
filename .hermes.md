# TOK Trading System — Quality Constraints

These constraints are NON-NEGOTIABLE. They apply to every Hermes session operating in this repo, every skill, every cron job, every output.

## Data Integrity — Paper Trading

- Every assertion about a position, order, or trade MUST be verified against live data (portfolio.json, ledger.json, or journal.db) before being reported. Never rely on memory or previous output.
- P&L figures MUST be independently verifiable from raw data. Show your working: `position_size × (current_price − entry_price)`.
- Never fabricate a value to fill a gap. If data is unavailable, say "data unavailable" — do not interpolate, guess, or synthesize.
- "I think", "I believe", "probably", "should be" are not acceptable for concrete facts (prices, positions, P&L). Use data or stay silent.
- The words `legacy_archived` in a trade's outcome means that trade's P&L MUST be excluded from all calculations. Filter it at every consumer.
- Ledger close sync is mandatory after every M2M cycle. `close_position()` return MUST include `trade_id`. Always save portfolio.json BEFORE ledger.json.

## Code Quality

- Before modifying any file in `scripts/` or `strategy/`, read the current state first. These files handle real data.
- After any code change touching P&L calculation, position management, or signal recording, run `python3 scripts/paper_trader.py --summary` to verify consistency.
- Never hardcode paths. Use `os.path.dirname(os.path.abspath(__file__))` patterns consistent with the existing codebase.
- When adding a new consumer of ledger/portfolio data, apply the `legacy_archived` filter.
- SQL queries in strategy_journal.py must use parameterized statements — never f-string interpolation in SQL.

## Audit Requirements

- After any trade open/close cycle, the portfolio ↔ ledger cross-reference MUST be consistent: symbols in `portfolio.positions` must match `status: "opened"` entries in ledger.
- Dangling signals in journal.db (signals with no outcome) MUST be investigated. If the corresponding paper trade doesn't exist, write an `orphaned` outcome.
- Any cron job that touches trading data must verify integrity before reporting results.

## Reporting

- When reporting trade results, always include: symbol, entry price, exit price, P&L %, and duration. Never report a partial picture.
- Separate analytical content from opinion. Label speculation explicitly with a confidence level.
- When uncertain about a data point, escalate to "data unavailable" — never paper over with prose.

## Violation Handling

If a violation of these constraints is detected (by self-check, delegated review, or cron audit), the response must:
1. Flag the specific violation with evidence
2. Correct the data or output
3. Document the root cause so it becomes a memory entry for future prevention
