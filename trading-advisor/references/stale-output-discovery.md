# Stale Output Discovery (2026-07-01)

## The Problem

The file `daily_briefing_compact_2026-07-01.md` (saved to disk by `briefing.py --compact --save-only`) contained **old code output** — ghost commands (`python3 scripts/paper_trader.py --summary`), news blocks, and enhanced regime data — while the **current code** had already fixed those issues. The file on disk was generated by a previous version of `briefing.py` and never overwritten.

Meanwhile, the **actual cron delivery** (what reached Telegram) used the current code and was correct. The disk file was a misleading artifact.

## Root Cause

`briefing.py`'s `save()` function (line 1910-1915) writes to `daily_briefing_compact_YYYY-MM-DD.md` every run. When the cron fires, it:
1. Runs `python3 scripts/briefing.py --compact --save-only --enhanced --orchestrator`
2. This runs the **current** code and both saves to disk AND returns text
3. The cron agent then delivers the text to Telegram

If someone later edits `briefing.py` (v3.2 fixes), the NEXT cron run would write fresh output. But if an investigator reads the disk file BEFORE the next cron, they see stale output.

## How to Detect

When investigating what the system actually delivers:

1. **Do not trust disk files alone.** Saved report files can be from prior code versions.

2. **Check the actual delivery.** For cron jobs, inspect the cron's `last_delivery` or the Telegram message history. The delivery output is what the user saw.

3. **Cross-reference key indicators.** These markers in the compact briefing distinguish old code vs current code:

| Indicator | Old Code (before v3.2) | Current Code |
|-----------|----------------------|--------------|
| Quick P&L line | `- Paper: python3 scripts/paper_trader.py --summary` | `- Paper: Equity $9,367.25 · Return 🟢 0.00%` |
| News block | Present (sentiment counts + headlines) | Not present |
| Halt notice | `🤷 No high-confidence setup today.` | `🚫 HALTED: All signals blocked by extreme market regime.` |
| Funnel summary | `📈 428 coins scanned → 170 ≥ $50M mcap → 60 ≥ 3% move` (no near-miss) | Same + near-miss if picks exist |

4. **Check the orchestrator digest.** With `--orchestrator`, current code saves to `orchestrator_digest_YYYY-MM-DD.md` and shows a reference in the briefing. Old code embedded the orchestrator block inline.

## Prevention

- Saved files should include a version/generation-hash comment so it's obvious when they're stale
- When investigating divergence, always compare disk files against a current `--dry-run` execution
- Cron deliveries are the source of truth for what the user actually received
